# Canary 01.5 — Multi-Operator Reduced-Economic Canary

**Status:** Closed (infrastructure validation passed; stake recovered 2026-05-26; archived)
**Network:** Base Sepolia (chainId 84532)
**Run window:** 2026-05-08 (single-day attended)
**Recovery date:** 2026-05-26 (all 5 stakes recovered)
**Profile:** `canary-01-5`

This document is the canonical record of the Canary 01.5 run. It is written for engineers who were not present during the canary and who need to understand what was tested, what was learned, what shipped to `main`, and what carries forward to Canary 02.

For the prior canary's findings see [`CANARY_01_RESULTS.md`](./CANARY_01_RESULTS.md). For the canary plan and its scenario matrix see [`CANARY_01_5.md`](./CANARY_01_5.md).

## Summary

Canary 01.5 was the first multi-operator canary run on `venom-node`. Its goal was to exercise five operator processes against a deployment with reduced economic constants (0.1 ETH minimum stake instead of 1.0 ETH; 3-of-5 quorum instead of 5-of-5) at a total testnet ETH budget under 1.0 ETH.

The infrastructure path validated end-to-end: configurable contract constants deployed and bound correctly on chain, the operator-environment pipeline generated five distinct operator wallets and matching docker-compose topology, the canary-readiness preflight gated cleanly at fund-up, and all five operators registered on chain successfully and began scanning blocks.

The live protocol path — five operators forming a libp2p gossip mesh, scoring a campaign, and submitting a `closeCampaign` aggregate signature — was **not exercised**. Two issues in the peer discovery path surfaced during the run: a separator regression in BullMQ queue naming (fully fixed) and a structural mismatch between on-chain registered multiaddrs and runtime libp2p peer identities (partially mitigated; full fix carried to Canary 02).

The protocol-level patches landed in this canary cycle (Modules A through F, plus a Module G partial) are the substantive output of the run.

## Validated infrastructure

### Module A — Configurable contract constants

`PilotEscrow` and `VenomRegistry` constructors accept profile parameters within bounded ranges (`require(...)` on every constructor argument). The same source code that deploys with production constants in production deploys with canary constants here, with **no source forks**. This was the first multi-profile deploy in the project.

A test fixture (`test/ConfigurableConstants.test.js`) verifies that `DEPLOY_PROFILE=canary-01-5` resolves to the expected constants tuple at deploy time and that out-of-bounds values revert with a clear error.

### Module B — Per-operator BullMQ queue isolation

Each operator runs:
- a scoped BullMQ queue (`venom-campaigns-op1` through `venom-campaigns-op5`)
- a scoped Redis cursor key (`venom:opN:producer:lastScannedBlock:0x...`)
- a scoped queued-campaign dedup key (`venom:opN:campaign:queued:<uid>`)

Multi-operator deployments against shared Redis no longer collide on jobId or producer cursor. The single-operator dev path (no `OPERATOR_QUEUE_SUFFIX` set) keeps the original unscoped names, preserving backward compatibility.

A regression test (`test/queue.isolation.test.js`) asserts both name shapes via `Queue` and `Worker` construction. The Worker construction assertion was added during this canary in response to Bug 1; see Findings below.

### Module C — Operator environment generator

`scripts/pilot/make-operator-envs.js --count=5 --profile=canary-01-5 --deployment=deployments/base-sepolia.json --out=.venom-canary/` produces:
- `manifest.json` (source-of-truth for downstream tooling)
- `funding-targets.txt` (faucet input list)
- `operator-N/.env` per operator (queue suffix, health port, bootstrap port, peer list, contract addresses)
- `docker-compose.canary-01-5.yml` (orchestration for the canary stack)

Security-property check passed: when the maintainer's shell has `DEPLOYER_PRIVATE_KEY` set, the generator emits `[WARN] DEPLOYER_PRIVATE_KEY is set in the calling environment and was intentionally not propagated` and writes nothing into operator envs that references it. This is asserted in `scripts/pilot/__tests__/make-operator-envs.test.js`.

Generated operator env files are written with `0600` permissions where the host filesystem honors POSIX modes. On NTFS via Git Bash they collapse to `0644` — acceptable on a single-user dev box, but documented as a Windows-host deviation in the canary evidence.

### Module D — Canary-readiness preflight (CIST phase 9)

The canary-readiness preflight phase reads the operator manifest and runs seven gates:

| Gate | Code | What it checks |
|---|---|---|
| Profile match | `CANARY_PROFILE_MISMATCH` | On-chain immutables match `profile.constants` |
| Deployment match | `CANARY_DEPLOYMENT_MISMATCH` | Manifest contract addresses match shell exports |
| Queue suffix validity | `CANARY_QUEUE_SUFFIX_INVALID` | Each suffix passes `aggregator/queue.js#normalizeQueueSuffix` |
| Operator-deployer collision | `CANARY_OPERATOR_IS_DEPLOYER` | No operator key equals the deployer key |
| Private multiaddr | `CANARY_PRIVATE_MULTIADDR` | No operator advertises a private multiaddr (escape hatch: `--local-only`) |
| Operator balance | `CANARY_OPERATOR_BALANCE_LOW` | Each operator wallet ≥ `MIN_STAKE + buffer` |
| Suffix uniqueness | (manifest property) | All 5 suffixes are distinct |

At canary fund-up the preflight returned 8 PASS + 1 WARN. The WARN was on the operator balance gate, where each wallet was funded to exactly the floor (0.16 ETH against a 0.16 floor). Conservative-by-design; no real risk to gas-budget headroom on Base Sepolia.

The legacy phase 7 (single-operator p2p preflight) returned FAIL because it derived its operator wallet from `OPERATOR_PRIVATE_KEY` (legitimately unset in canary mode) and reported its absence. Documented as MF-5 in the backlog; phase 7's gates should skip when `--canary-envs` is active.

### Module E — Regression test suite

12 MAIN-FIX entries from the Canary 01 backport tracker now have regression tests. The suite caught Bug 1 (the BullMQ queue name regression) within 30 seconds of attempted reproduction.

Pytest plumbing was added for ML service security tests; `ml_service/tests/test_api_key.py` covers two cases (sentinel rejection in testnet mode, allowed in non-testnet) — both passing.

### Module F — Multi-container orchestration

The canary stack runs 7 containers from a single compose file:
- 1 × `venom-redis-canary` (Redis 7-alpine, ACL'd, mapped to host port 6380)
- 1 × `venom-ml-service-canary` (FastAPI + sentence-transformers, mapped to host port 8001)
- 5 × `venom-node-canary-N` (operator processes; no host port mapping, only internal bridge networking)

Both supporting services run with healthchecks. Operator containers depend on both being healthy before they start, so registration and producer scanning don't begin until the data plane is fully up.

Distinct host port mappings (6380 vs 6379, 8001 vs 8000) prevent collision with the dev stack defined in `docker-compose.yml`. The two stacks can coexist on the same machine.

### MultiRPC failover

`RPC_URLS` was upgraded from 2 to 3 providers during the canary in response to Base Sepolia public RPC flakiness (`sepolia.base.org` returning 502 Bad Gateway intermittently). The MultiRPC layer (`rpc/router.js`) correctly falls over between providers with exponential backoff. The canary's final `RPC_URLS`:

```
RPC_URLS=https://base-sepolia.gateway.tenderly.co,https://base-sepolia-rpc.publicnode.com,https://sepolia.base.org
```

### DNS in containers

Docker Desktop's embedded DNS resolver lost its upstream during one canary session, surfacing as `getaddrinfo ENOTFOUND sepolia.base.org` errors in the producer logs even though host-side DNS was working. Mitigation: explicit `dns: [1.1.1.1, 8.8.8.8]` per venom-node service in the canary compose file. The dev compose file does not need this on most environments; canary's larger inter-container fan-out exposed it.

## Findings

### Bug 1 — BullMQ queue name colon regression (MF-6, **fixed**)

**Symptom.** All 5 operator containers crash-looped with `Error: Queue name cannot contain :` from BullMQ's `Worker` constructor, 12 seconds after first stack-up.

**Root cause.** Module B introduced operator-scoped queue names in `aggregator/queue.js`:

```javascript
return normalizedSuffix ? `${baseName}:${normalizedSuffix}` : baseName;
```

BullMQ 5+ rejects `:` in queue names. The `Queue` constructor's validator was lenient and accepted the name; the `Worker` constructor's validator rejected it. The producer side never threw. The worker side threw on construction, and `register_and_start.js` exited the container, which compose then restarted, looping indefinitely.

**Why the existing test missed it.** `test/queue.isolation.test.js` exercised `Queue` construction but not `Worker` construction. The two BullMQ classes share most validation but not all of it.

**Fix.** Single-character change in `aggregator/queue.js`:

```diff
-return normalizedSuffix ? `${baseName}:${normalizedSuffix}` : baseName;
+return normalizedSuffix ? `${baseName}-${normalizedSuffix}` : baseName;
```

`operatorScopedKey()` (Redis-key namespacing) keeps `:` because Redis itself imposes no separator restriction; BullMQ's restriction is on the queue name only.

**Test gap closed.** Added `Worker` construction assertion to `test/queue.isolation.test.js` — instantiates a `Worker` with `autorun: false` for each valid suffix and asserts the constructor doesn't throw.

**Backward compatibility.** Single-operator dev mode is unaffected. With no suffix, the name is just `venom-campaigns` (no separator at all). The hyphen change is invisible to anyone not running multi-operator.

### Bug 2 — Stale on-chain multiaddrs (MF-7, **partial fix shipped; full fix carried to Canary 02**)

**Symptom.** Post-Bug-1-fix, operators logged `[P2P] Failed to dial /ip4/172.22.0.4/tcp/42047/p2p/12D3KooWMbeh...` repeatedly. All 5 operators, all dialing the same peer ID and the same address. Mesh did not form.

**Root cause, traced through three layers.**

1. The repo-root `.env` had a stale `PUBLIC_MULTIADDR` line from a prior single-operator dev run (peer ID `12D3KooWMbeh...`, port 42047). It had been latent for unknown duration.
2. When the canary stack first came up, all 5 fresh containers inherited the same `PUBLIC_MULTIADDR` from the repo-root `.env` (which is loaded as a fallback by every operator on top of its operator-specific env). Each called `registerOracle()` with that string. The on-chain registry now permanently has 5 oracles, each registered with an identical, dead multiaddr.
3. `VenomRegistry` has no method to update an oracle's multiaddr after registration. Correcting requires a new contract deployment, which would orphan the existing 0.50 ETH stake commitments.

The runtime libp2p peer ID changes every container restart (no key persistence), so even the *peer ID portion* of `PUBLIC_MULTIADDR` was wrong from the moment the env was inherited. The multiaddr was effectively never valid for this canary.

**Partial mitigation (Module G partial; shipped).**

The `aggregator/p2p.js` dial loops now honor `VENOM_SKIP_REGISTRY_DIAL=true` and skip the registry-derived dials entirely. mDNS peer discovery was added via `@libp2p/mdns` for an alternative discovery path. Tested mDNS — it does not bridge across Docker Desktop's bridge network (multicast UDP is not reliably forwarded). Added `P2P_LISTEN_PORT` (deterministic per-operator listen port, `42001..42005`) and `P2P_BOOTSTRAP_PEERS` (comma-separated `host:port` list of sibling operators) so canary mesh formation can use container-name DNS + fixed ports rather than mDNS or the broken registry. The bootstrap path uses `/dns4/<service-name>/tcp/<port>` multiaddrs, dialing without specifying a peer ID — libp2p learns the peer's identity inline during the noise handshake.

The full fix and the live mesh evidence both belong to Canary 02. See "Carried forward."

**Why this happened and how to prevent it.** A single uncommented `PUBLIC_MULTIADDR=` line in the repo-root `.env` poisoned five fresh wallets' worth of on-chain state. The repo-root `.env` should not contain operator-specific runtime values. The `.env.example` in this repo will be revised in a follow-up doc PR to make the operator-env vs deploy-env separation explicit.

### Bug 3 — mDNS does not bridge on Docker Desktop's bridge network (informational; mitigation shipped)

**Symptom.** Module G's mDNS path was added to `aggregator/p2p.js` (`peerDiscovery: [mdns()]`) and verified as physically loaded in each running container. After 60 seconds of uptime, no operator had any peer-to-peer TCP connections — only Redis (`:6379`) and external HTTPS (`:443` to RPC providers).

**Root cause.** Docker Desktop on Windows does not reliably forward multicast UDP across its bridge network. mDNS depends on multicast broadcasts to advertise and discover peers on the local segment. Without working multicast, mDNS is silent.

**Mitigation.** The Module G partial bypassed mDNS for canary mesh formation by using direct DNS-resolved bootstrap dials (see Bug 2). The mDNS path is retained in code as a no-op for this canary's environment, but functions in production-style cross-host topologies where multicast routing is configured (or where alternative discovery is in place).

**For Canary 02.** This finding doesn't generalize beyond Docker Desktop on Windows. Linux Docker (where most production deploys run) does forward multicast across bridges. The mDNS fallback should still be evaluated on the actual Canary 02 environment before deciding to remove it.

## What was not validated

These were canary scenario goals that were not exercised before the run was wound down. They carry forward to Canary 02 in priority order.

1. **Live multi-operator gossip mesh.** Five operators forming a libp2p gossipsub mesh and exchanging signed score messages. The bootstrap-peers + pinned-ports path was implemented and source-tested but not observed forming a live mesh.
2. **Live `closeCampaign` aggregate.** A campaign opened on `PilotEscrow`, scored by the quorum-required subset of operators (3 of 5 in this canary's profile), with the leader submitting an aggregate signature that mints the on-chain close event.
3. **Live abstain matrix.** The original plan's matrix of "all 5 score / 4 score + 1 abstain / 3 score + 2 abstain" scenarios. The 3+2 case is the most informative — it exercises the participation floor at the exact threshold (3 = `REQUIRED_ORACLES`, 5 active = 60% participation, 60 ≥ 67 floor: actually fails by 7 points, meaning the close path correctly rejects).
4. **Live deviation/slashing observation.** A scoring divergence above `MAX_DEVIATION` triggering an `OracleSlashed` event. The contract-side slashing path is proven in `test/PilotEscrow.v1.1.0.test.js` test 4c, but live evidence has not yet been produced in any canary. (This was also a gap in Canary 01.)

## Carried forward to Canary 02

**Must-haves before Canary 02 fund-up:**

1. **Full Module G.** Persistent libp2p keys per operator (write at env generation, read at runtime), AND a `VenomRegistry.updateOracleMultiaddr()` method (requires contract redeploy with the change), AND a one-time correction of registered multiaddrs. Specification work begins from the Module G partial that shipped in this canary; the runtime side already accepts deterministic `P2P_LISTEN_PORT` and reads `P2P_BOOTSTRAP_PEERS`, so persistent keys are the missing piece.

2. **Live mesh evidence.** The bootstrap-peer + pinned-port path needs to be observed forming a real gossipsub mesh under load, with peer-to-peer TCP connections visible per operator and at least one campaign's worth of cross-operator score gossip. This was the biggest gap in Canary 01.5 and is the first acceptance gate for Canary 02.

3. **Cross-host networking.** At least 2 of 5 operators on a separate host (physical, VM, or distinct Docker host on a different machine). Validates that the bootstrap-peers path works across actual networks rather than only across a Docker bridge. Pure-Linux Docker hosts may also show mDNS working (multicast forwarding is more reliable on Linux than on Docker Desktop), which would change the discovery story.

4. **Live scenario execution.** The full canary scenario matrix from `CANARY_01_5.md` (sections "Scenarios A-D"), with the abstain matrix executed deterministically. Requires a working mesh.

**Nice-to-haves before Canary 02 (none are blockers):**

- MF-3, MF-4, MF-5, MF-8 — small fixes that didn't ship in 01.5 cycle.
- A consolidated `CANARY_02.md` plan document mirroring the structure of `CANARY_01_5.md` but with revised scenarios reflecting what was learned here.
- A revised `.env.example` that explicitly partitions deploy-time vs operator-runtime variables, to prevent the class of bug behind MF-7.

## Verdict

Canary 01.5's stated goal: *"Reduced-economic multi-operator canary close to 1 total testnet ETH capacity, validating the protocol-level patches landed across Modules A through F."*

**On goal.** Modules A through F are validated. Five operators stake, register, scan, and report active on-chain. Queue isolation works per operator. Configurable constants deploy and verify on-chain. The preflight + manifest + envgen pipeline functions end-to-end. Multi-container orchestration works. RPC failover works. Container DNS resolution works (with overrides). The infrastructure substrate that Canary 02 will run on is validated and stable.

**Not on goal.** Live mesh formation and live campaign close were not observed. The path forward is concrete: Module G full (persistent libp2p keys + registry update method) plus cross-host validation. The Module G partial that shipped in this canary cycle is implemented and tested at source level — it is not yet validated under live load.

**On budget.** ~0.96 ETH committed against 1.0 ETH ceiling; ~0.013 ETH non-recoverable. Within scope.

The protocol-level work that landed (Modules A through F, plus Module G partial, plus the MF-6 fix) is the substantive output of this canary. The mesh-discovery limitation surfaced here is structural, not cosmetic, and would have surfaced in any equivalent multi-operator canary on Docker Desktop with the existing libp2p discovery configuration. Canary 02 inherits a clean scope of work and a clear acceptance target.
